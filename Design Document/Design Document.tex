\section{Introduction}

\subsection{Purpose}
The purpose of this document is to describe the architecture, components, interfaces, and detailed design of the \textbf{University Timetable Management System}. This document translates the Software Requirements Specification into technical design decisions that guide the system implementation and its  verification.

\subsection{Project Overview}
The University Timetable Management System is web based application that aims to automate scheduling and management of timetables. Instead of manual and busy-schedules it substitutes it with centralized timetable creation, conflict detection and timetable access services to students, faculty, administrators.

\subsection{Design Approach}

The deisgn of our system is based on these principles:
\begin{itemize}
\item Independent modules and interfaces with clarity.
\item Every design element traces back to SRS requirements.
\item Interface designs prioritize ease of use.
\end{itemize}

\section{Design Assumptions and Constraints}

\subsection{Design Assumptions}
\begin{itemize}
    \item Users have access to modern web browsers.
    \item Academic, course, and user data is provided by the university.
    \item Email services are available for notifications.
    \item User roles and permissions are predefined.
    \item The system operates for a single university.
\end{itemize}

\subsection{Design Constraints}
\begin{itemize}
    \item Compliance with university academic and data privacy policies.
    \item Deployment limited to university IT infrastructure.
    \item Must support concurrent users during peak periods.
    \item Secure handling of sensitive user information.
\end{itemize}


\section{Key Design Decisions}

\subsection{Architectural Decisions}
The University Timetable Management System is developed based on a layered, modular.
architecture which is consistent with the system context and interaction of components stipulated in the SRS. The architecture is based on a three-tier architecture:

\begin{itemize}
    \item \textbf{Presentation Layer}: A web-based user interface that was developed towards various.
    user roles (Admin, Faculty, Student). This layer handles
    user interaction, form validation, timetable visualization and dashboard rendering.
    
    \item \textbf{Application Logic Layer}: This layer fulfills fundamental system capability.
    such as schedule creation, schedule editing, schedule conflict detection, schedule conflict resolution proposals, schedule publishing, and schedule notification. Business rules are
    enforced at this layer.
    
    \item \textbf{Data Access Layer}: In responsibility of communication with the relational database.
    This layer deals with permanent storing, retrieval and updating of timetables, users,
    Audit logs, courses, locations and system configurations.
\end{itemize}

This level of architectural separation encourages scalability, maintainability and testability, and is.
shown in the sequence diagram.

\subsection{Database Design Decisions}
According to the database specifications as per the SRS, a relational database management.
There is a choice of system (PostgreSQL or MySQL). The database schema is created through normalized.
relations (up to Third Normal Form) to decrease redundancy and guarantee the consistency of data.

Key design decisions include:
\begin{itemize}
    \item Separate tables for users, roles, courses, department, venue, timetables.
    \item Enforcement of relationships between entities  using foreign key constraints, such as
    course, lecturers and venue.
    \item Soft-delete of timetables and user accounts to agree with business, audit requirements and rules.
\end{itemize}


\subsection{Security Design Decisions}
Security design decisions are derived by the non-functional security requirements defined in
the SRS. The system implements Role-Based Access Control (RBAC) to ensure that users can
only perform actions allowed according to their roles.

Key security design elements include:
\begin{itemize}
    \item Authentication using secure login credentials with password hashing algorithm.
    \item Audit logging of critical actions such as login attempts, timetable modifications,
    and publishing events.
    \item Secure session management with timeout mechanisms
    access.
\end{itemize}


\subsection{User Interface Design Decisions}
The user interface design follows a role-based approach aligned with the user classes
identified in the SRS. Each user role is provided with a his/her dashboard according to its
responsibilities.

Key UI design decisions include:
\begin{itemize}
    \item \textbf{Admin Dashboard}: Provides access to timetable creation, conflict reports,
    user management, and system configuration.
    \item \textbf{Faculty Dashboard}: Displays assigned courses, timetable view, and
    umakeup class functionality.
    \item \textbf{Student Dashboard}: Displays a read only personalized timetable view.
\end{itemize}

Timetables are presented using calendar-style for ease of use. These design decisions
are demonstrated.
